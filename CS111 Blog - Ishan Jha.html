🗂️ Data Types — The Foundation

Every value in code is one of five fundamental data types. The type isn't just a label—it determines what operations are valid and how logic behaves. Mixing types incorrectly breaks comparisons and calculations.

TypeExamplesWhat It Does
Number 5, 3.14, -42, Infinity Arithmetic, comparisons, timing. health = 12, moveSpeed = 1.4, Date.now() timestamp.
String 'Daddy Pig', "laser.png", `${x}` File paths, messages, template literals. src: `${path}/images/${name}.png`
Boolean true, false State flags, conditions. isDefeated, isVulnerable, battleEnded
Array [1, 2, 3], [], [obj1, obj2] Collections of values. this.lasers = [], this.classes = [...]
Object {x: 10, y: 20}, {health: 12, speed: 1.4} Structured data. Config objects, laser state, position coordinates.

Why Type Matters

// Type determines what you can do with a value
5 + 3                // Numbers: arithmetic works → 8
"5" + "3"            // Strings: concatenation → "53"
5 + "3"              // Mixed: JavaScript coerces → "53" (string)
 
if (health <= 0)    // Health is a Number; comparison works
if (isDefeated)        // isDefeated is Boolean; gates logic
for (const L of lasers)  // lasers is Array; iteration works
Every conditional check, every loop, every calculation silently depends on type. When you write `if (count > 5)`, the `>` operator expects two numbers. If `count` is a string, JavaScript might coerce it (converting `"10" > 5` to `true`), but that's not reliable. Understand your data's type first.

🔢 Operators — The Computations

Operators are what make code actually do something. Every frame, operators run the arithmetic, comparisons, and logic that produce the game's behavior. Master these and you master computation.

CategoryOperatorsExample & Meaning
Mathematical + - * / % ** health - damage (subtraction), position.x += vx (add and assign), base ** exp (power)
Comparison == === != !== < > <= >= health === 0 (exact match), time >= cooldown (at or past threshold)
Logical && || ! x && y (both true), x || y (at least one true), !x (invert)

Real Example: Collision Detection with Operators

// AABB Collision: four comparisons with && (all must be true)
if (laserLeft         <  bossRight     &&    // laser left inside boss right
    laserRight        >  bossLeft      &&    // laser right past boss left
    laserTop          <  bossBottom    &&    // laser top inside boss bottom
    laserBottom       >  bossTop) {            // laser bottom past boss top
  // All FOUR conditions true → collision confirmed
  boss.takeDamage(1);
  this.lasers.splice(i, 1);
}
 
// If ANY condition is false, the whole && chain short-circuits to false
// This is why operator order matters for performance
The four < and > operators check rectangle overlap on both axes. The && operator chains them—all four must be true simultaneously for a hit. If the laser flew past the boss entirely, one comparison fails and the whole condition is false. No collision detected. This is the foundation of game collision systems.

🔁 Nested Conditionals — Complex Decision Making

A single if statement makes a binary choice. Nested conditionals layer multiple checks, letting you handle complex scenarios: "If the player pressed the button AND enough time has passed AND the boss is still alive, then fire a laser."

Why Nesting Matters

Without nesting, you'd need many separate checks. With nesting, each layer narrows the scope, making logic clearer and preventing invalid state transitions. 

// BAD: flat logic — unclear flow and redundant checks
if (attackRequested) { }
if (cooldownElapsed) { }
if (bossAlive) { }
// What if cooldown passed but boss is dead? Logic unclear.
 
// GOOD: nested logic — clear guard structure
if (attackRequested) {                      // Outer check: action requested
  if (cooldownElapsed && bossAlive) {      // Inner check: conditions met
    spawnLaser();
  }
}
Nested conditionals create a "guard clause" pattern: layer by layer, invalid states are filtered out. The attack only happens if ALL conditions pass. This prevents bugs where a laser fires even though the boss is dead, or the cooldown hasn't elapsed.

Real Example: Boss Damage System

takeDamage(amount = 1) {
  // Nested Condition 1: Is the boss already dead?
  if (this.isDefeated) {
    console.log('Already defeated. Ignore.');
    return false;  // Exit early — no damage to dead boss
  }
  
  // Nested Condition 2: Is the boss currently invulnerable?
  if (!this.isVulnerable) {
    console.log('Invulnerable window active. Block hit.');
    return false;  // Exit early — hit is blocked
  }
  
  // All outer guards passed. Now apply damage.
  this.health -= amount;
  this.isVulnerable = false;             // Start 300ms cooldown
  this.lastDamagedAt = Date.now();
  
  // Nested Condition 3: Did this damage defeat the boss?
  if (this.health <= 0) {
    this.isDefeated = true;
    this.canvas.style.filter = 'grayscale(1) brightness(0.8)';
    console.log('Boss defeated!');
    return true;
  }
  
  return true;  // Damage was successfully applied
}
This method demonstrates three nested checks at different points:
  • Check 1: Guard clause at start — if already defeated, exit immediately.
  • Check 2: Invulnerability window check — prevent same laser doing damage twice in 300ms.
  • Check 3: After damage applied — if health drops to zero, trigger defeat state.
Each nesting layer narrows scope, making the logic impossible to misunderstand. A laser can never damage a dead boss because of Check 1. A laser can never hit twice in rapid succession because of Check 2. And defeat is confirmed only after damage, not before.

Attack Cooldown Example: Nested Conditions + Operators

updateAttack(now) {
  ///csse/javascript/constellators/conditionals-hw Outer: Did player press attack this frame?
  if (this.attackRequested) {
    this.attackRequested = false;  // Consume the input
    
    // Inner Layer 1: Has enough time passed since last attack?
    if (now - this.lastAttackTime >= this.attackCooldown) {
      
      // Inner Layer 2: Is the boss still alive?
      if (!this.boss.isDefeated) {
        this.lastAttackTime = now;
        this.spawnLaser();
        console.log('Laser fired!');
      }
    }
  }
}

// Call this every frame
game.updateAttack(Date.now());
Operator breakdown:
  • now - this.lastAttackTime >= this.attackCooldown: Subtract numbers, then compare with >= operator. True only if enough time has passed.
  • !this.boss.isDefeated: The ! (NOT) operator inverts the boolean. Only fire if boss is NOT defeated.
  • this.attackRequested = false: Assignment operator consumes the input so it doesn't fire again this frame.
The nested structure ensures the attack fires only if all three conditions align: input requested → cooldown expired → boss alive.

📐 OOP — Classes, Inheritance & Constructor Chaining

Class Hierarchy (2+ levels deep)

The game is organized into two inheritance chains. The first controls the boss enemy — what it looks like, how it moves, and how it takes damage. The second controls each level — what objects appear, what the background is, and what rules the battle follows. Without this hierarchy, every level would need its own copy of all the laser, HUD, and collision code.

GameObject ← engine base (provided) └── Enemy ← sprite, canvas, position (provided) └── PeppaBossEnemy ← MY CLASS: health, movement, damage
PeppaBattleLevelBase ← MY BASE CLASS: laser system, HUD, collisions ├── PeppaLevel1 ← George Pig config ├── PeppaLevel2 ← Peppa Pig config └── PeppaLevel3 ← Daddy Pig (final boss) config

Writing Custom Classes

PeppaBossEnemy is the class that controls everything about the boss character during a fight — how much health it has, whether it's currently alive, how fast it chases the player, and whether it can be hit right now. Without this class existing as a separate object, there would be no boss to fight. The constructor runs once when the level starts and sets up all the starting values that control boss behavior for the entire fight. 

/**
 * PeppaBossEnemy — custom boss class extending the Enemy base.
 * Adds health system, defeat state, invulnerability window, and hitbox debug.
 * @class PeppaBossEnemy
 * @extends Enemy
 */
class PeppaBossEnemy extends Enemy {
 
  /**
   * @param {Object} data     - Sprite and config data for this enemy
   * @param {Object} gameEnv  - Shared game environment (canvas, objects, path)
   */
  constructor(data = null, gameEnv = null) {
    super(data, gameEnv);        // constructor chaining → Enemy → GameObject
 
    this.health         = data?.health    ?? 3;   // Number
    this.maxHealth      = this.health;
    this.moveSpeed      = data?.moveSpeed ?? 0.8; // Number
    this.isDefeated     = false;                  // Boolean
    this.isVulnerable   = true;                   // Boolean
    this.showHitbox     = false;                  // Boolean
    this.lastDamagedAt  = 0;                      // Number (timestamp)
    this.invulnerabilityMs = 300;               // Number (cooldown window)
 
    console.log(`[PeppaBossEnemy] Spawned: ${data?.id} with ${this.health} HP`);
  }
}
super(data, gameEnv) is what makes the boss actually appear on screen. Calling it runs the Enemy constructor, which sets up the canvas element and loads the boss sprite image from the src path. Without that super() call, the boss would have no image, no canvas to draw on, and no position in the game world — it simply wouldn't exist visually. Everything my custom code adds (health, speed, defeat state) layers on top of what super() builds.

🗂️ Data Types

Every piece of game state is stored as one of five data types. The type isn't just a label — it determines what operations are valid and how the game logic behaves with that value. Using the wrong type would break the comparisons and calculations that make the game work.

TypeHow It Controls the GameCode Example
Number health tracks how many more hits the boss can take before the level ends. moveSpeed is how many pixels the boss moves toward the player each frame — 1.4 for Daddy Pig vs 1.0 for George Pig is what makes him feel faster and harder. Date.now() timestamps control every cooldown timer in the game. this.health = 5 · this.moveSpeed = 1.4 · Date.now() · life: 60
String enemyImage is the file path the engine uses to load the boss sprite — get this string wrong and the boss appears as a broken image. Template literals build the HUD messages shown on screen. enemyName: 'Daddy Pig' · 'grayscale(1)' · template literals

Boolean Expression Example — AABB Collision Detection

// All FOUR conditions must be true simultaneously with && for a collision hit
if (hitLeft              <  boss.position.x + boss.width  &&   // left edge check
    hitLeft + hitW        >  boss.position.x               &&   // right edge check
    hitTop               <  boss.position.y + boss.height  &&   // top edge check
    hitTop  + hitH        >  boss.position.y) {                  // bottom edge check
  boss.takeDamage(1);
  this.lasers.splice(i, 1);
}
These four conditions together check whether a laser rectangle and the boss rectangle overlap on both axes at the same time. If the laser has passed the boss's left edge (hitLeft < boss.x + boss.width), hasn't gone past the right edge, is below the top, and above the bottom — all four true simultaneously means the rectangles intersect and the laser hit. If any one condition is false — for example the laser flew past the boss entirely — the && chain short-circuits and no damage is dealt. This is what makes hit detection in the game feel spatially accurate rather than registering hits at arbitrary distances.

🔁 Control Structures

Iteration — for loop over laser array

// Iterate backwards through the lasers array — safe to splice while looping
for (let i = this.lasers.length - 1; i >= 0; i--) {
  const L = this.lasers[i];
 
  L.x    += L.vx;        // move laser each frame
  L.y    += L.vy;
  L.life -= 1;           // count down lifespan
 
  if (L.life <= 0) {
    this.lasers.splice(i, 1);   // remove expired laser
    continue;
  }
  // ... collision checks below ...
}
This loop is what makes all lasers fly across the screen simultaneously each frame. Without it, a laser would be spawned and immediately frozen — nothing would update its position. The backwards iteration matters because when a laser expires or hits something, it gets removed from the array with splice(i, 1). If you loop forward and remove index 3, what was at index 4 slides down to index 3 and gets skipped that frame — meaning a laser could pass through the boss without registering a hit. Looping backwards means only already-visited indices shift, so nothing is skipped.

Conditionals — state transitions in the game loop

update() {
  // Conditional 1: guard clause stops execution if battle is over
  if (this.battleEnded) return;
 
  // Conditional 2: check win condition
  if (boss.isDefeated && !this.battleEnded) {
    this.battleEnded = true;
    const isFinalLevel = ctrl.currentLevelIndex === ctrl.levelClasses?.length - 1;
    if (isFinalLevel) {
      this.showWinScreen();
    } else {
      this.messageTimeout = setTimeout(() => {
        if (ctrl?.currentLevel) ctrl.currentLevel.continue = false;
      }, 1100);
    }
    return;
  }
 
  // Conditional 3: lose condition
  if (this.playerHealth <= 0 && !this.battleEnded) {
    this.battleEnded = true;
    this.showLoseScreenAndRestart();
  }
}

Nested Conditionals — 3-level depth example in boss damage logic

// Level 1: Check if damage was requested by collision system
if (laserHitBoss) {
  // Level 2: Prevent overkill damage on already-defeated boss
  if (!boss.isDefeated) {
    // Level 3: Respect 300ms invulnerability cooldown window
    if (Date.now() - boss.lastDamagedAt >= 300) {
      boss.takeDamage(1);
    }
  }
}
This three-layer nesting is essential for fair combat mechanics. The outermost condition gates the whole system — if no collision happened, nothing runs. The middle level prevents a defeated boss from taking additional damage and showing wrong behavior on-screen. The innermost level is the cooldown check — without it, a single laser frame passing through the boss would trigger damage on every frame update while overlapped, instantly killing the boss. The three conditions together create realistic, predictable game behavior.

🎮 Practical Examples: From Game Code

All three core concepts — nested conditionals, operators, and data types — appear together in real game code. Here's how they work as a unified system:

The BossGame Attack System (Complete Example)

// Data types: Number, Boolean, Object
class BossGame {
  constructor() {
    this.boss = {
      health: 10,                    // Number
      isDefeated: false,           // Boolean
      isVulnerable: true,          // Boolean
      lastDamagedAt: Date.now(),   // Number (timestamp)
      invulnerabilityMs: 300        // Number (cooldown window)
    };
  }

  takeDamage(amount = 1) {
    // Operators: !== (comparison), -= (assignment)
    if (this.boss.health !== 0) {           // comparison operator
      this.boss.health -= amount;             // assignment operator
    }
    
    // Nested Conditional (3 levels)
    if (!this.boss.isDefeated) {                // NOT operator, Level 1
      if (this.boss.isVulnerable) {           // Level 2
        if (Date.now() - this.boss.lastDamagedAt >= this.boss.invulnerabilityMs) {
          this.boss.lastDamagedAt = Date.now();
          console.log(`Boss HP: ${this.boss.health}`);
        }
      }
      // Check defeat condition
      if (this.boss.health <= 0) {           // comparison operator
        this.boss.isDefeated = true;      // Boolean assignment
      }
    }
  }
}
This example shows all three concepts working together:
  • Data Types: Number (health, timestamps), Boolean (state flags), Object (boss config)
  • Operators: Arithmetic (+=, -=), Comparison (<=, >=, !==), Logical (&&, !)
  • Nested Conditionals: Three levels deep controlling when damage applies and when boss is defeated
